# Save Tears Web 管理端设计规范

## 1. 设计定位

Web 管理端延续小程序居民端已经落定的 `精品酒店刊物` 视觉语言，但信息密度更高、更偏运营视角。它不是冰冷的监控大屏，也不是纯品牌海报，而是一个“前台背后的服务台”：

- 同样使用浅水色、雾白材质、深海墨蓝作锚点色。
- 同样保持中文为主、少量英文点缀的排版语气。
- 但布局更规整、密度更高、列表与筛选更靠前。

这套规范只定义未来 Web 管理端该如何被实现，不在本轮创建 `save_tears_frontend` 代码。

## 2. 设计令牌

### 2.1 颜色

- `Ink / 深海墨蓝`：`#173843`
- `Ink Soft / 深青蓝`：`#234954`
- `Mist / 雾白`：`#F6FBFB`
- `Mist Soft / 海盐白`：`#EDF5F4`
- `Panel / 玻璃白`：`rgba(255,255,255,0.72)`
- `Line / 细边线`：`rgba(27,64,75,0.12)`
- `Text / 主文本`：`#16343D`
- `Text Soft / 次文本`：`#55707A`
- `Text Muted / 提示文本`：`#789099`
- `Warning / 暖棕提示`：`#915D3B`
- `Warning Soft / 暖棕底`：`#F4E4D9`
- `Success / 稳定绿`：`#2C6B5D`

### 2.2 排版

- 封面级标题：高对比 serif，中文优先使用宋体气质字体或品牌定制标题字体。
- 页面级标题：serif，字重 600 左右，行高 1.05-1.12。
- 正文与表格：现代 sans-serif，重点保持清晰和稳定，不要过度压缩。
- 英文只做栏目语气或状态标签，不承担主要信息传达。

### 2.3 材质与阴影

- 大部分模块采用轻玻璃感白底，不使用厚重投影。
- 阴影以大面积柔影为主，避免“卡片漂浮过高”的廉价感。
- 深色块只用于关键指标、主 CTA、顶部控制台或需要强锚点的区域。

## 3. 页面布局规则

### 3.1 整体框架

- 顶部为固定刊物式页眉：页面 kicker、主标题、当前日期区间或筛选摘要。
- 左侧为可选导航栏，右侧为主体内容区；主体采用 `摘要带 -> 操作区 -> 列表 / 图表` 的顺序。
- 页面第一屏永远先给“管理判断信息”，不是先给长表格。

### 3.2 摘要带

- 每个关键页面都需要一条摘要带，放 3-4 个 KPI。
- KPI 卡片不做等权重复；至少保留 1 个深色锚点卡，其余使用浅色面板。
- 摘要带下面再进入筛选、表格、列表或趋势图。

### 3.3 列表与表格

- 列表行高要足够呼吸，不要做成高压密度表格。
- 主字段左对齐，辅助字段与状态 badge 集中在右侧。
- 表格头必须弱化，不要大面积深底；优先用浅底细线。

## 4. 关键组件规则

### 4.1 KPI 卡

- 包含 `标签 / 值 / 一句解释` 三层。
- 值使用 serif 数字或高对比数字排版。
- 解释语尽量是判断句，不是原始字段名重复。

### 4.2 搜索与筛选条

- 搜索条采用长圆角胶囊型，玻璃白背景。
- 筛选项用分段胶囊或下拉，不要使用传统密集按钮阵列。
- 搜索和筛选永远放在摘要带之后，不要抢第一页注意力。

### 4.3 状态标签

- `管理员`：深海墨蓝底白字。
- `住户`：浅雾白底深字。
- `待补房间号`：暖棕浅底。
- `正常映射`：浅青灰底。

### 4.4 趋势图

- 趋势图优先使用细线或轻柱图，不使用高饱和色。
- 只有异常点允许用暖棕或更深的色值强调。

## 5. 关键页面蓝图

### 5.1 总览首页

**目标**
让管理员在 5 秒内理解：当前用户规模、角色分布、房间映射是否完整，以及今天有没有明显异常。

**布局**
- 顶部刊物式页眉：`Admin Desk / 住户总览`
- 第一排摘要卡：总用户数、管理员数、已绑房间数、待补房间数
- 第二排：搜索 / 筛选条
- 第三排：住户列表预览
- 第四排：缺失房间号名单或重点提醒

**交互**
- 搜索支持用户名、房间号双条件模糊查询
- 缺失房间号可点击进入详情页或侧边抽屉

### 5.2 住户目录页

**目标**
管理端批量查看所有住户的基础信息状态。

**布局**
- 页眉显示筛选摘要，如“全部住户 / 已绑房间 / 近 30 天活跃”
- 列表区以卡片列表或浅表格混合呈现
- 行内展示：用户名、房间号、角色、最近数据同步状态

**交互**
- 支持按角色、房间绑定状态、是否有异常记录筛选
- 行点击进入住户详情页

### 5.3 住户详情页

**目标**
把单个住户的小程序数据体验翻译为管理视角。

**布局**
- 顶部：住户身份摘要卡
- 中段：用水、账单、浊度三段切换
- 底部：最近提醒、房间映射状态、备注区

**交互**
- 支持快速切换三类数据
- 支持标记备注、记录人工跟进状态

### 5.4 异常与提醒页

**目标**
集中处理高峰用水、浊度异常、房间信息缺失等需要人工关注的项目。

**布局**
- 顶部：异常数量摘要带
- 中部：按异常类型分段
- 底部：异常条目列表，支持批量标记已处理

**交互**
- 每条异常进入后可查看住户详情
- 支持筛选“未处理 / 已处理 / 今日新增”

## 6. 与小程序的一致性要求

- 小程序首页和 Web 管理端首页必须共享同一套颜色语气和标题语言。
- 小程序的“数据中心”与 Web 的“住户详情页”应遵循相同的数据分段顺序：流量、账单、浊度。
- 管理端虽然更紧凑，但绝不切回传统蓝白后台模板。

## 7. 实施建议

- 先实现 `总览首页 + 住户目录页 + 住户详情页`，再扩展异常与提醒页。
- 组件库层先落 `KPI 卡 / 搜索条 / 状态标签 / 浅表格 / 趋势图卡`。
- 如果后端仍然只有 `/users`，则 Web 第一阶段也应遵循这次小程序管理员页的过渡版原则，不伪造不存在的数据能力。
